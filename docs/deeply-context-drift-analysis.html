Deeply 回归测试解释页

为什么问“科学史入门书”，它却继续讲“拖延”？

这次现象不是 OpenClaw 连接断了，也不是 Deeply 搜索配置没生效。更准确地说，是 Deeply 的“推荐课程”触发规则把一句新的明确需求误路由到了“根据刚才整段对话推荐课程”的 prompt。

一句话结论

P2 体验问题

你的输入里有“推荐”，所以客户端把它判定为“我要基于当前 Deeply 对话生成推荐卡”。但这条专用 prompt 又明确要求模型“回顾刚才整段对话”。由于这个会话之前主要在聊拖延，模型就按旧主题给出了皮尔斯·斯蒂尔和《拖延心理学》。

用户以为自己在问：给我一本科学史入门书。
系统实际暗示模型：用户点了推荐课程按钮，请从刚才整段对话里提炼推荐方向。

实际发生了什么

1. Deeply 会话前文在聊“拖延” 页面上已有“拖延心理学 / 皮尔斯·斯蒂尔”相关内容。
2. 你输入“推荐一本适合入门的科学史书” 这是一个新主题，并且目标很明确：科学史、入门书、一本。
3. 客户端关键词命中“推荐” Deeply 当前规则看到“推荐”就容易进入推荐卡路径。
4. outbound builder 换成推荐卡专用 prompt 用户看到的还是原句，但真正发给 OpenClaw 的文本被包了一层“推荐课程”指令。
5. 推荐卡 prompt 要求“回顾刚才整段对话” 模型自然回到了前文最强上下文：拖延。
6. UI 渲染了拖延相关推荐卡 所以看起来像“没听懂科学史”，其实是路由和 prompt 共同把它拉回了旧主题。

为什么不是搜索配置问题

连接链路是通的

同一轮测试里 Koko 普通聊天、酒馆角色聊天、Deeply 返回推荐卡都成功了，没有 websocket 或 provider 报错。

返回内容是“有逻辑地错”

如果是搜索配置失败，通常会报工具不可用或没有结果。这次它生成了完整推荐卡，只是主题被旧上下文带偏。

关键判断：它不是“不会搜索科学史”，而是压根没有被稳定地引导到“科学史入门书”这个新任务上。

代码层面的触发点

触发规则太宽

shouldTriggerDeeplyRecommend 目前会把包含“推荐”的短句判定为推荐卡路径。你的句子“推荐一本适合入门的科学史书”长度短，并且包含“推荐”，所以命中。 

miniapps/deeply/mobile/persona.ts:119-133

const DEEPLY_RECOMMEND_TRIGGER_REGEX =
  /(推荐|列|来一?组|换一组|再来|清单|学习计划|课题清单|课程清单)/;

export function shouldTriggerDeeplyRecommend(visibleText: string): boolean {
  if (visibleText === DEEPLY_RECOMMEND_VISIBLE_TEXT) return true;
  const text = visibleText.trim();
  if (text.length === 0 || text.length > 60) return false;
  if (!DEEPLY_RECOMMEND_TRIGGER_REGEX.test(text)) return false;
  if (!DEEPLY_RECOMMEND_HARD_HINT_REGEX.test(text)) return false;
  return true;
}

推荐路径强制“回顾刚才整段对话”

命中后，Deeply 不走普通聊天 prompt，而是走 DEEPLY_RECOMMEND_INSTRUCTION。这里写的是“用户点击了推荐课程按钮”，并要求“回顾我们刚才整段对话”。 

miniapps/deeply/mobile/persona.ts:1208-1217

[系统注入 · 用户点击了「推荐课程」按钮]

请回顾我们刚才整段对话,从中提炼 3-5 个最值得深入学习的方向,
然后把推荐结果作为一个 fenced block 返回。

每个方向应满足:
- 在对话里被实际提到过,或与对话主题强相关。

用户看见的文本和模型收到的文本不同

这是设计上有意为之：用户看到的是原句，OpenClaw 收到的是包装后的 gatewayText。这让 UI 很干净，但也意味着 intent routing 一旦错了，模型会被隐藏 prompt 带偏。 

miniapps/deeply/mobile/index.ts:97-120

if (shouldTriggerDeeplyRecommend(visibleText)) {
  return {
    visibleText,
    gatewayText: [
      DEEPLY_RECOMMEND_INSTRUCTION,
      "",
      "[用户消息]",
      visibleText
    ].join("\n")
  };
}

问题大不大

不算阻塞 它不影响连接、流式、卡片渲染、OpenClaw 调用，也不说明搜索 provider 配错。
体验要修 Deeply 是探索产品，用户换主题时如果还抓着旧主题，会明显降低信任感。
beta 前建议 不需要大重构，主要修 intent routing 和推荐 prompt 优先级。

建议怎么修

方案 A：拆分按钮推荐和文字推荐

只有用户真的点“定制课程 / 推荐课程”按钮时，才使用“回顾整段对话”的推荐卡 prompt。

方案 B：文字推荐优先当前句

如果用户文字里有明确新主题，比如“科学史入门书”，prompt 必须把这个新主题作为最高优先级。

方案 C：调窄关键词规则

“推荐一本书 / 推荐一篇文章 / 推荐资料”不要自动进课程卡路径，除非同时出现“课程、课题、深入学习、几门”。

方案 D：加主题切换判定

如果当前用户消息和前文主题差异明显，就明确告诉模型“这是新主题，不要沿用上一轮主题”。

最小可行修法：保留按钮路径不动；用户手打“推荐…”时改成一个新 prompt，写清楚“当前用户消息优先，只有用户说‘刚才/这个/继续/再推荐’才回顾旧对话”。

快速判断

根因：推荐 intent 被误触发，加上推荐 prompt 强调旧对话。

不是：OpenClaw 掉线、搜索 provider 缺失、Deeply API 不通。

影响：用户换主题时可能答偏；不会破坏基础通信。

优先级：P2，beta 前最好修。
